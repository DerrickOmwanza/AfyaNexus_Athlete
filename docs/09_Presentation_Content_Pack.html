Presentation Strategy

The presentation should be concise, visual, and tightly aligned to the institution’s progress stage assessment criteria. Since the guideline emphasizes methodology and design, the slide sequence below gives that section the largest share of the five-minute window. The recommended presentation is 7 slides.

Slide 1: Title and Project Identity
Approx. 20–25 seconds
  • Project title: AfyaNexus: Centralized Athlete Training, Recovery, Nutrition, and Injury Risk Prediction System
  • Derrick Omwanza Atandi - I231/G/4534/22 and Geoffrey Ooko - I231/G/4510/22
  • Programme: ITB 1410 Project
  • One-line identity statement: “AfyaNexus integrates athlete logs, wearable data, and AI-assisted injury-risk analysis into one platform.”
Slide 2: Problem Statement and Project Definition
Approx. 35–40 seconds
  • Athlete data is fragmented across notebooks, chat apps, wearables, and disconnected staff workflows.
  • This makes it difficult to monitor readiness, nutrition, and injury-risk in a unified way.
  • The project develops a centralized multi-role system for athletes, coaches, and nutritionists.
  • Core innovations: hybrid data collection, wearable and Android health synchronization, and AI-supported prediction.
Slide 3: Justification
Approx. 30–35 seconds
  • Relevant to AI, mobile health, wearable computing, and digital sports analytics.
  • Benefits athletes through better visibility into training, recovery, and nutrition.
  • Benefits coaches and nutritionists through shared, role-specific dashboards.
  • Supports evidence-based and timely decision-making.
Slide 4: Literature, Prior Work, and Technology Context
Approx. 30 seconds
  • Prior work shows athlete monitoring is strongest when training, recovery, and nutrition are considered together.
  • Commercial ecosystems provide partial solutions, but AfyaNexus centralizes these functions in one academic prototype.
  • Official technology foundations include web frameworks, Supabase, Python ML services, and Android Health Connect.
Slide 5: Methodology and Design
Approx. 70–80 seconds
  • Methodology: Agile for incremental system development + CRISP-DM-inspired process for data and prediction workflow.
  • Architecture: Athlete / Android Phone / Wearable → Frontend / Android Bridge → Express API → Supabase → Flask injury-risk service.
  • Design techniques: role-based architecture, relational schema, REST APIs, dashboard analytics, explainable injury-risk scoring.
  • Data sources: manual logs, wearable sync, Android Health Connect import.
  • Quality control: validation, source labels, and ambiguity detection for manual versus synced data.

Suggested visual: Use one architecture diagram or flowchart here. This should be your strongest slide.

Slide 6: Current Progress and Prototype Evidence
Approx. 50–55 seconds
  • Implemented athlete, coach, and nutritionist dashboards.
  • Implemented training, recovery, nutrition, wearable, and diet-plan workflows.
  • Implemented baseline injury-risk prediction service.
  • Added Android Health Connect import endpoint and Android bridge scaffold.
  • Added coach/nutritionist assignment flow and source labels for synced versus manual records.

Suggested visual: 2–3 screenshots of dashboard, sync flow, or athlete reports.

Slide 7: Project Plan, Risks, and Next Step
Approx. 40–45 seconds
  • Resources: laptop, Android phone, internet/LAN connection, Supabase backend, Android Studio, test users.
  • Risks: mobile permissions, demo network issues, ambiguous data, limited labeled injury dataset.
  • Mitigation: validation, fallback manual demo, preloaded screenshots, LAN testing before presentation.
  • Immediate next step: complete live Android-to-AfyaNexus sync and final-stage testing/polish.
