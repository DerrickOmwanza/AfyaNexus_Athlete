1. Architecture Overview

AfyaNexus follows a layered, hub-and-spoke architecture. The athlete is the central data source (hub), while coaches and nutritionists are data consumers (spokes). All data flows through a central database and machine learning processing layer before being surfaced on role-specific dashboards. Access to all layers is governed by Role-Based Access Control (RBAC).

Core Model: Athlete (Data Hub) → Central DB + ML Layer → Role-Based Dashboards (Coach / Nutritionist)

2. System Layers

Layer 1 — Data Collection Layer
  • Manual input: Athlete fills morning check-in (sleep, soreness, mood) and post-training log (workout type, intensity, duration).
  • Automated input: wearable devices or health platforms sync activity, heart rate, sleep duration, step count, and workout summaries.
  • Android-first adapter: Google Health Connect on Android acts as the first synchronized source demonstrated in AfyaNexus.
  • Hybrid nutrition input: diet data is collected either manually or through connected nutrition ecosystems such as mobile health or diet-logging applications.
Layer 2 — Central Database Layer
  • Stores all athlete profiles, training logs, recovery logs, wearable data, nutrition logs, diet plans, and injury predictions.
  • Enforces data relationships: Athlete ↔ Coach (one-to-many), Athlete ↔ Nutritionist (one-to-many).
Layer 3 — Machine Learning Processing Layer
  • Receives combined training and recovery datasets from the central database.
  • Applies data-quality checks to detect ambiguous or conflicting manual versus wearable entries before prediction is generated.
  • Processes patterns to generate an injury risk score (0–100) and risk level (Low / Medium / High).
  • Outputs predictions back to the central database for dashboard display.
Layer 4 — Presentation Layer (Role-Based Dashboards)
  • Athlete Dashboard: Personal data, wearable sync status, injury risk gauge.
  • Coach Dashboard: Sponsored athletes' performance, injury alerts, training suggestions.
  • Nutritionist Dashboard: Registered athletes' nutrition logs, diet plan tools, performance-nutrition charts, and data-quality review.
Layer 5 — Security & Access Control Layer (RBAC)
  • Registration assigns each user a role: Athlete, Coach, or Nutritionist.
  • Authentication (login) verifies identity and grants role-specific dashboard access.
  • Coaches can only view athletes they sponsor; nutritionists can only view athletes registered under them.

3. Entities and Relationships

Entity Role Relationship
Athlete Primary data source; owns and inputs all personal data Can have one Coach and one Nutritionist
Coach Monitors and guides athlete training Can sponsor many Athletes (one-to-many)
Nutritionist Monitors and guides athlete nutrition Can register many Athletes (one-to-many)
Wearable / Health Platform Automated capture of training and recovery metrics Linked to one Athlete through Bluetooth, mobile health sync, or vendor APIs

5.2 Data Captured by T70

  • Heart rate (real-time and average)
  • Sleep duration and sleep quality metrics
  • Step count and activity logs

5.3 Integration Flow

The T70 device connects to the athlete's mobile device or system interface via Bluetooth. A sync API endpoint (POST /api/wearable/sync) receives the parsed device data and stores it in the WearableData table. This data is then combined with manual inputs for ML processing.
