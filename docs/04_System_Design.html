1. Functional Requirements

# Requirement Description
FR-01 Athlete Registration Athletes can register with name, email, password, and role. They are linked to a coach and nutritionist upon registration or later.
FR-02 Morning Check-In Athletes input sleep hours, soreness level (1–10), mood, and any numbness or discomfort each morning.
FR-03 Post-Training Log Athletes log workout type, intensity (1–10), duration, and any post-training notes after each session.
FR-04 Wearable Device Sync The system syncs with supported wearable devices or health platforms to automatically capture heart rate, sleep duration, steps, and activity data.
FR-05 Injury Risk Prediction The ML model processes training and recovery data to generate an injury risk score (0–100) and risk level (Low / Medium / High).
FR-06 Athlete Dashboard Athletes view their personal logs, wearable sync status, and injury risk score on their dashboard.
FR-07 Coach Dashboard Coaches view performance data, injury alerts, and training trends for athletes they sponsor only.
FR-08 Nutritionist Dashboard Nutritionists view nutrition logs and create diet plans for athletes registered under them only.
FR-09 Role-Based Access Control Each user role (Athlete, Coach, Nutritionist) has restricted access to only their authorized data and dashboards.
FR-10 Secure Authentication All users authenticate via email and password. Sessions are managed using JWT tokens.
FR-11 User Profile & Settings Management All users (Athletes, Coaches, Nutritionists) can view and update their profile name, email, and password from a dedicated Settings page within their portal.
FR-12 Hybrid Nutrition Data Collection Nutrition intake can be captured manually by the athlete or imported from connected diet and health applications, then reviewed by the nutritionist.
FR-13 Data Quality Validation The system validates all manual and synced data, flags ambiguous entries, and highlights conflicts between manual logs and wearable-derived metrics.

2. Non-Functional Requirements

# Requirement Target
NFR-01PerformanceDashboard load time under 2 seconds; ML prediction response under 3 seconds.
NFR-02ScalabilitySupport up to 500 concurrent users without performance degradation.
NFR-03SecurityAll passwords hashed with bcrypt; all API endpoints secured with JWT; data encrypted at rest.
NFR-04UsabilityIntuitive UI with consistent color scheme; mobile-responsive design.
NFR-05ReliabilitySystem uptime of 99% or higher.
NFR-06Data IntegrityAll inputs validated server-side; wearable and nutrition sync data cross-checked against manual entries.

3. Database Schema

The AfyaNexus database uses a relational model (PostgreSQL). Below are the core tables and their key fields.

3.1 Athletes Table

CREATE TABLE Athletes ( id SERIAL PRIMARY KEY, name VARCHAR(100) NOT NULL, email VARCHAR(150) UNIQUE NOT NULL, password_hash TEXT NOT NULL, specialization VARCHAR(100), -- e.g., 100m, marathon coach_id INT REFERENCES Coaches(id), nutritionist_id INT REFERENCES Nutritionists(id), created_at TIMESTAMP DEFAULT NOW() );

3.2 Coaches Table

CREATE TABLE Coaches ( id SERIAL PRIMARY KEY, name VARCHAR(100) NOT NULL, email VARCHAR(150) UNIQUE NOT NULL, password_hash TEXT NOT NULL, created_at TIMESTAMP DEFAULT NOW() );

3.3 Nutritionists Table

CREATE TABLE Nutritionists ( id SERIAL PRIMARY KEY, name VARCHAR(100) NOT NULL, email VARCHAR(150) UNIQUE NOT NULL, password_hash TEXT NOT NULL, created_at TIMESTAMP DEFAULT NOW() );

3.4 TrainingLogs Table

CREATE TABLE TrainingLogs ( id SERIAL PRIMARY KEY, athlete_id INT REFERENCES Athletes(id) NOT NULL, date DATE NOT NULL, workout_type VARCHAR(100), -- e.g., sprint, endurance intensity INT CHECK (intensity BETWEEN 1 AND 10), duration_min INT, notes TEXT, created_at TIMESTAMP DEFAULT NOW() );

3.5 RecoveryLogs Table

CREATE TABLE RecoveryLogs ( id SERIAL PRIMARY KEY, athlete_id INT REFERENCES Athletes(id) NOT NULL, date DATE NOT NULL, sleep_hours DECIMAL(4,2), soreness_level INT CHECK (soreness_level BETWEEN 1 AND 10), mood VARCHAR(50), -- e.g., Good, Tired, Anxious numbness BOOLEAN DEFAULT FALSE, notes TEXT, created_at TIMESTAMP DEFAULT NOW() );

3.6 WearableData Table

CREATE TABLE WearableData ( id SERIAL PRIMARY KEY, athlete_id INT REFERENCES Athletes(id) NOT NULL, date DATE NOT NULL, heart_rate_avg INT, sleep_duration DECIMAL(4,2), steps INT, device_id VARCHAR(50), -- T70 Bluetooth address synced_at TIMESTAMP DEFAULT NOW() );

3.7 InjuryPredictions Table

CREATE TABLE InjuryPredictions ( id SERIAL PRIMARY KEY, athlete_id INT REFERENCES Athletes(id) NOT NULL, date DATE NOT NULL, risk_score DECIMAL(5,2), -- 0.00 to 100.00 risk_level VARCHAR(20), -- Low / Medium / High model_ver VARCHAR(20), created_at TIMESTAMP DEFAULT NOW() );

3.8 NutritionLogs Table

CREATE TABLE NutritionLogs ( id SERIAL PRIMARY KEY, athlete_id INT REFERENCES Athletes(id) NOT NULL, date DATE NOT NULL, calories INT, protein_g DECIMAL(8,2), carbs_g DECIMAL(8,2), fats_g DECIMAL(8,2), meal_notes TEXT, created_at TIMESTAMP DEFAULT NOW() );
Optional future extension for clinical nutrition / biomarker integration NutritionBiomarkers ( id SERIAL PRIMARY KEY, athlete_id INT REFERENCES Athletes(id) NOT NULL, collected_at TIMESTAMP NOT NULL, source_system VARCHAR(100), -- lab, EMR, clinician upload calcium_level DECIMAL(8,2), iron_level DECIMAL(8,2), vitamin_d_level DECIMAL(8,2), notes TEXT );

3.9 DietPlans Table

CREATE TABLE DietPlans ( id SERIAL PRIMARY KEY, nutritionist_id INT REFERENCES Nutritionists(id) NOT NULL, athlete_id INT REFERENCES Athletes(id) NOT NULL, plan_name VARCHAR(150), recommendations TEXT, created_at TIMESTAMP DEFAULT NOW(), updated_at TIMESTAMP DEFAULT NOW() );

4. API Endpoints

4.1 Authentication

MethodEndpointDescription
POST/api/auth/registerRegister a new user (Athlete, Coach, or Nutritionist) with role assignment.
POST/api/auth/loginAuthenticate user and return JWT token with role-based dashboard redirect.
GET/api/auth/onboarding-optionsReturn available coaches and nutritionists for athlete onboarding and assignment.
POST/api/auth/logoutInvalidate the current session token.

4.2 Athlete APIs

MethodEndpointDescription
POST/api/athlete/sources/health-connect/importImport normalized Android Health Connect data for recovery, wearable metrics, training, and nutrition logs.
GET/api/athlete/dashboardRetrieve athlete's personal dashboard data (logs + risk score).
POST/api/athlete/recovery-logSubmit morning check-in (sleep, soreness, mood, numbness).
POST/api/athlete/training-logSubmit post-training log (workout type, intensity, duration).
POST/api/athlete/nutrition-logSubmit daily nutrition log (calories, protein, carbs, fats, meal notes).
GET/api/athlete/reportsRetrieve 30-day performance and health trend data for the athlete reports page.
GET/api/athlete/wearableRetrieve registered device ID and last 10 wearable sync entries.
POST/api/athlete/wearable/syncSync T70 wearable data (heart rate, sleep, steps) to the athlete's profile.

4.3 Coach APIs

MethodEndpointDescription
GET/api/coach/athletesList all athletes sponsored by the authenticated coach.
GET/api/coach/dashboard/:athlete_idView full performance dashboard for a specific sponsored athlete.
GET/api/coach/injury-alertsRetrieve injury risk alerts for all sponsored athletes.

4.4 Nutritionist APIs

MethodEndpointDescription
GET/api/nutritionist/athletesList all athletes registered under the authenticated nutritionist.
GET/api/nutritionist/athletes/:athlete_id/dashboardView nutrition logs and performance data for a specific registered athlete.
GET/api/nutritionist/diet-plansRetrieve all diet plans across all athletes registered under the nutritionist.
POST/api/nutritionist/diet-planCreate a new diet plan for a registered athlete.

4.5 Settings & Profile API

MethodEndpointDescription
GET/api/auth/profileRetrieve the authenticated user's profile (name, email, role).
PUT/api/auth/profileUpdate the authenticated user's name and/or email.
PUT/api/auth/passwordUpdate the authenticated user's password (requires current password verification).

4.6 ML Prediction API

MethodEndpointDescription
POST/api/ml/predictTrigger injury risk prediction for a specific athlete using their latest training and recovery data.
GET/api/ml/history/:athlete_idRetrieve historical injury risk predictions for an athlete.

5. User Interface Design Notes

5.1 Color Palette

RoleColorHex CodePurpose
PrimaryDeep Blue#1E3A8AHeaders, navigation, primary buttons
SecondaryEmerald Green#10B981Success states, health indicators
AccentWarm Orange#F97316Alerts, highlights, injury risk indicators
BackgroundSoft Gray#F3F4F6Page backgrounds, card backgrounds
TextCharcoal Black#111827Body text, labels

5.2 Athlete Dashboard Layout

  • Top Bar: Athlete profile, T70 sync status indicator, notification bell.
  • Sidebar: Dashboard, Training Log, Recovery Log, Nutrition Log, Wearable Sync, Reports, Settings.
  • Main Area:
    • Sleep Duration Card (hours + quality indicator)
    • Morning Check-In Card (soreness, mood, numbness)
    • Workout Summary Card (type, intensity, duration)
    • Injury Risk Gauge (visual 0–100 score with color-coded risk level)

5.3 Coach Dashboard Layout

  • Top Bar: Coach profile, notification bell.
  • Sidebar: Dashboard, My Athletes, Settings.
  • Settings Page: Update profile name, email, and password.
  • Main Area:
    • Athlete List (with injury risk badges)
    • Performance Overview Graph (training load over time)
    • Injury Risk Alerts Panel (color-coded: green/orange/red)
    • Training Suggestions Section

5.4 Nutritionist Dashboard Layout

  • Top Bar: Nutritionist profile, notification bell.
  • Sidebar: Dashboard, My Athletes, Diet Plans, Settings.
  • Settings Page: Update profile name, email, and password.
  • Main Area:
    • Athlete List (with nutrition compliance indicators)
    • Nutrition Logs Table (meal type, description, calories)
    • Diet Plan Creator (form to add/edit recommendations)
    • Performance-Nutrition Correlation Chart
